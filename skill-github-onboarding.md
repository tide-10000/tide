# github-onboarding

帮助非编码者完成在 GitHub 上的第一次参与——从注册账号到创建第一条 Issue。

---

## 触发条件

当用户表达以下意图时激活此技能：

- 「我不懂 GitHub」「我不会编程，怎么参与」
- 「我想报告一个问题但不知道怎么做」
- 「我有一个想法想告诉开发者」
- 「我有经验想分享，但不会用技术工具」
- 任何表现出「想参与开源但对 GitHub 感到恐惧或困惑」的信号

---

## 工作流

### 步骤 1：确认场景

识别用户属于哪种场景：

| 用户意图 | 引导方向 |
|:---|:---|
| 报告问题 | → 教程第四章 · 模板 1（Bug 报告） |
| 提功能建议 | → 教程第四章 · 模板 2（功能需求） |
| 分享经验 | → 教程第四章 · 模板 3（经验贡献） |
| 参与讨论 | → 教程第四章 · 模板 4（参与讨论） |
| 成为服务商 | → 教程第四章 · 模板 5（申请成为服务商） |
| 完全不知道 GitHub 是什么 | → 从教程第一章开始 |

### 步骤 2：按需引导

根据用户的知识水平选择起点：

- **零基础（不知道 GitHub 是什么）**→ 从第一章「GitHub 是什么」开始
- **有账号但没用过 Issue**→ 跳到第四章「创建你的第一条 Issue」
- **需要模板**→ 直接提供对应的模板，帮用户填写

### 步骤 3：帮助填写

对于不会格式化内容的用户：

1. 用对话的方式收集信息（一问一答）
2. 将用户的回答填入对应的 Issue 模板
3. 把填好的模板发给用户确认
4. 引导用户复制粘贴到 GitHub Issue 页面

### 步骤 4：引导提交

帮助用户完成最后的提交步骤：

1. 确认用户在项目页面（发给用户链接）
2. 指导用户点击「New Issue」
3. 指导用户粘贴标题和正文
4. 指导用户点击「Submit new issue」

---

## 关键原则

- **不说术语**：用比喻代替专业术语（仓库 = 图书馆的书，Issue = 留言簿）
- **分步指引**：每次只说一步，用户完成后再给下一步
- **允许犯错**：明确告诉用户「Issue 可以随时编辑，写错了可以改」
- **用中文**：整个引导过程使用中文
- **确认理解**：每一步完成后简短确认（不是考用户，是确认自己讲清楚了）
